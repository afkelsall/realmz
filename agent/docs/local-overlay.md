# Local-only files: the `gitl` overlay repo

This checkout carries developer-local files that must **never** end up in the Realmz
repo's history or in a PR to upstream: `CLAUDE.md`, the `agent/` docs, the `scripts/`
build helpers (`build-windows.sh`, `rebuild-integration.sh`, `TC-mingw.cmake`,
`integration.txt`), and `branches.md`. They have to live in the working tree to be
useful (the build scripts run from here, `CLAUDE.md` is read on session start), but
they are not part of the project.

They are kept in a **second git repo overlaid on the same working directory** - the
classic "bare dotfiles repo" trick.

## How it is wired

- **Overlay git dir:** `~/Development/realmz-local.git` (bare; work tree is the Realmz
  checkout at `~/Development/realmz`).
- **Branch:** `local-tooling`, pushed to the fork `afkelsall/realmz` as an
  unrelated-history branch (it shares no commits with `main` or any feature branch).
- **Alias:** `gitl` runs git against that overlay. It is defined in `~/.bashrc`:

  ```sh
  alias gitl='git --git-dir=$HOME/Development/realmz-local.git --work-tree=$HOME/Development/realmz'
  ```

- **`status.showUntrackedFiles no`** is set on the overlay so `gitl status` does not
  list the entire Realmz tree as untracked.
- The same files are listed in **`.git/info/exclude`** of the *Realmz* repo, so the
  normal `git` never sees them and they cannot be staged onto a feature branch.

The two repos share one folder but not one history: normal `git` operates on Realmz,
`gitl` operates on the local files only.

## The rule: overlay vs. a real Realmz branch

Decide by asking "should this ever reach upstream?"

- **Goes in the overlay (`gitl`):** anything developer-local or agent-facing -
  `CLAUDE.md`, `agent/` docs, the `scripts/` build/integration helpers,
  `integration.txt`, `branches.md`, personal tooling. These are excluded from the
  Realmz repo by design.
- **Goes in a normal Realmz branch (`git`):** any change to the actual game or shim -
  `src/`, `src/realmz_orig/`, `CMakeLists.txt`, `README.md`, vendored submodules,
  game data. These become PRs and must follow the contribution and "no AI marker"
  rules in `CLAUDE.md`.

If you are unsure, it almost certainly belongs in a real branch, because the overlay
is only for things that must stay out of upstream.

## Everyday use

```sh
gitl status                 # only the local files
gitl add scripts/foo.sh     # stage a local change (always name explicit paths)
gitl commit -m "..."        # plain message, no AI markers (the branch is public on the fork)
gitl push                   # back up to origin/local-tooling on the fork
gitl pull                   # pull updates on another machine
```

Two cautions:

- **Never `gitl add -A` or `gitl add <dir>`.** The overlay has no ignore rules for the
  Realmz tree, so a broad add would try to stage the whole project. Always add explicit
  file paths.
- The `local-tooling` branch is **publicly visible** on the fork. Keep commit messages
  plain and marker-free, same as the project convention in `CLAUDE.md`.

## Adding a new local file

1. Create it in the working tree.
2. `gitl add <path>` and commit it to the overlay.
3. Add the path to the Realmz repo's `.git/info/exclude` so normal `git` keeps ignoring
   it. Use a blanket entry only for directories the Realmz repo does not track; for a
   shared directory like `scripts/` (which the Realmz repo tracks
   `scripts/extract_mac_metadata.py`) list each local file individually so genuinely new
   upstream files still show up.

## Restore on a new machine

After cloning the Realmz repo normally:

```sh
git clone --bare https://github.com/afkelsall/realmz.git ~/Development/realmz-local.git
gitl config status.showUntrackedFiles no
gitl checkout local-tooling      # populates CLAUDE.md, agent/, scripts/, branches.md
```

Then re-add the local entries to `.git/info/exclude`.

## What is NOT in the overlay

Build output and fetched dependency sources - `build_win/`, `build_win_debug/`,
`.deps-src/` - are regenerated by the build and are not versioned anywhere. They are
listed in `.git/info/exclude` purely to keep `git status` clean.
